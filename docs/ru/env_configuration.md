# Конфигурация переменных окружения

## Работа с .env.example и .env

Скопируйте `.env.example` в `.env` и заполните необходимые значения. Файл `.env` не должен попадать в git (уже в .gitignore). Все переменные читаются из `.env` при старте приложения через pydantic-settings.

## Группы переменных

### Application

**APP_ENV**
- Что делает: Окружение приложения (development/production)
- Когда заполнять: Всегда
- Значение по умолчанию: `development`
- Как проверить: В логах видно окружение

**APP_TIMEZONE**
- Что делает: Часовой пояс для временных меток
- Когда заполнять: Если нужен другой часовой пояс
- Значение по умолчанию: `Asia/Yerevan`
- Как проверить: Проверьте временные метки в логах и базе данных

### Market Data Providers

**OANDA_API_KEY**
- Что делает: API ключ OANDA для получения свечей Forex
- Когда заполнять: Обязательно для работы с рынком
- Как получить: Зарегистрируйтесь на oanda.com, в личном кабинете создайте API токен
- Права: Доступ к практике (practice) или реальному счету
- Как проверить: Запустите анализ, проверьте что свечи загружаются

**OANDA_ACCOUNT_ID**
- Что делает: ID счета OANDA
- Когда заполнять: Обязательно вместе с OANDA_API_KEY
- Как получить: В личном кабинете OANDA в разделе Account
- Как проверить: Вместе с проверкой OANDA_API_KEY

**OANDA_BASE_URL**
- Что делает: Базовый URL API OANDA
- Когда заполнять: Только если используете другой endpoint
- Значение по умолчанию: `https://api-fxpractice.oanda.com` (practice)
- Как проверить: Не требует проверки, используется автоматически

**TWELVE_DATA_API_KEY**
- Что делает: API ключ Twelve Data (fallback провайдер свечей)
- Когда заполнять: Опционально, для fallback если OANDA недоступен
- Как получить: Зарегистрируйтесь на twelvedata.com, в Dashboard создайте API key
- Права: Free tier достаточно для базового использования
- Как проверить: Если OANDA недоступен, должен использоваться автоматически

**TWELVE_DATA_BASE_URL**
- Что делает: Базовый URL API Twelve Data
- Когда заполнять: Только если используете другой endpoint
- Значение по умолчанию: `https://api.twelvedata.com`
- Как проверить: Не требует проверки

**RUNTIME_ENV**

- Когда заполнять: Обязательно, определяет какие провайдеры использовать
- Значение по умолчанию: `local`
- Как проверить: В логах должны использоваться провайдеры из выбранной ветки

### LLM Router Config

**LLM_ROUTER_MODE**
- Что делает: Режим роутера (sequential или strict)
- Когда заполнять: Только если нужен strict режим (не рекомендуется)
- Значение по умолчанию: `sequential`
- Как проверить: 
  - sequential: Fallback провайдеры должны использоваться по порядку
  - strict: Используется только primary провайдер/модель, без fallback и last resort (только одна попытка)

**LLM_VERIFIER_ENABLED**
- Что делает: Включить/выключить верификацию рекомендаций
- Когда заполнять: Если нужна верификация вывода
- Значение по умолчанию: `false`
- Как проверить: При `true` должна выполняться верификация после synthesis

**LLM_VERIFIER_MODE**
- Что делает: Режим верификации (soft или hard)
- Когда заполнять: Если включена верификация
- Значение по умолчанию: `soft`
- Как проверить: При hard должны выполняться попытки исправления

**LLM_VERIFIER_MAX_REPAIRS**
- Что делает: Максимальное количество попыток исправления
- Когда заполнять: Если включена верификация в режиме hard
- Значение по умолчанию: `1`
- Как проверить: В логах должны быть попытки repair

**LLM_MAX_RETRIES**
- Что делает: Максимальное количество повторных попыток для одного провайдера
- Когда заполнять: Если нужны другие настройки retry
- Значение по умолчанию: `3`
- Как проверить: В логах должны быть попытки retry при ошибках

**LLM_TIMEOUT_SECONDS**
- Что делает: Глобальный таймаут для всех LLM вызовов
- Когда заполнять: Если нужен другой базовый таймаут
- Значение по умолчанию: `60.0`
- Как проверить: В логах должен использоваться указанный таймаут

**LLM_TEMPERATURE**
- Что делает: Глобальная температура для всех LLM вызовов
- Когда заполнять: Если нужна другая креативность ответов
- Значение по умолчанию: `0.2`
- Как проверить: Ответы должны быть более/менее детерминированными

### Task Routing

**LLM_LAST_RESORT_PROVIDER**
- Что делает: Провайдер для last resort fallback
- Когда заполнять: Если нужен другой last resort
- Значение по умолчанию: `ollama_local`
- Как проверить: При недоступности всех провайдеров должен использоваться этот

**LLM_LAST_RESORT_MODEL**
- Что делает: Модель для last resort fallback
- Когда заполнять: Если нужна другая модель для last resort
- Значение по умолчанию: `llama3:latest`
- Как проверить: В логах last resort должен использовать эту модель

Переменные для каждой задачи следуют паттерну `{TASK}_{BRANCH}_{TYPE}_PROVIDER` и `{TASK}_{BRANCH}_{TYPE}_MODEL`, где:
- TASK: TECH, NEWS, SYNTHESIS, VERIFIER
- BRANCH: LOCAL, SERVER
- TYPE: PRIMARY, FALLBACK1, FALLBACK2, FALLBACK3

Примеры: `TECH_LOCAL_PRIMARY_PROVIDER`, `NEWS_SERVER_FALLBACK1_MODEL`.

### Logging

**LOG_DIR**
- Что делает: Директория для файлов логов
- Когда заполнять: Если нужна другая директория
- Значение по умолчанию: `logs`
- Как проверить: Файлы логов должны создаваться в указанной директории

**LOG_LEVEL**
- Что делает: Уровень логирования (DEBUG, INFO, WARNING, ERROR)
- Когда заполнять: Если нужен другой уровень детализации
- Значение по умолчанию: `INFO`
- Как проверить: В логах должен быть указанный уровень

**LOG_CONSOLE_LEVEL**
- Что делает: Уровень логирования в консоль
- Когда заполнять: Если нужен другой уровень для консоли
- Значение по умолчанию: `INFO`
- Как проверить: В консоли должны быть только сообщения указанного уровня

**LOG_FORMAT**
- Что делает: Формат логов (json или text)
- Когда заполнять: Если нужен другой формат
- Значение по умолчанию: `json`
- Как проверить: Логи должны быть в указанном формате

**LOG_ROTATION**
- Что делает: Время ротации логов (например, 00:00 для ежедневной ротации в полночь)
- Когда заполнять: Если нужно другое время ротации
- Значение по умолчанию: `00:00`
- Как проверить: Логи должны ротироваться в указанное время

**LOG_HTTP_LEVEL**
- Что делает: Уровень логирования HTTP библиотек
- Когда заполнять: Если нужен другой уровень для HTTP логов
- Значение по умолчанию: `WARNING`
- Как проверить: HTTP логи должны быть на указанном уровне

**LOG_SPLIT_FILES**
- Что делает: Разделять логи по файлам (app.log, errors.log, warnings.log)
- Когда заполнять: Если нужны отдельные файлы
- Значение по умолчанию: `true`
- Как проверить: Должны создаваться отдельные файлы логов

**LOG_ENABLE_HTTP_FILE**
- Что делает: Включить отдельный файл для HTTP логов
- Когда заполнять: Если нужны отдельные HTTP логи
- Значение по умолчанию: `false`
- Как проверить: Должен создаваться http.log

**LOG_MASK_AUTH**
- Что делает: Маскировать секреты в логах
- Когда заполнять: Всегда должен быть включен
- Значение по умолчанию: `true`
- Как проверить: В логах не должно быть видно API ключей

**LOG_RETENTION**
- Что делает: Время хранения логов
- Когда заполнять: Если нужен другой период
- Значение по умолчанию: `30 days`
- Как проверить: Старые логи должны удаляться

**LOG_COMPRESSION**
- Что делает: Сжатие старых логов (zip или gz)
- Когда заполнять: Если нужен другой формат
- Значение по умолчанию: `zip`
- Как проверить: Старые логи должны сжиматься

## Таймауты

Приоритет таймаутов (от высшего к низшему):

1. Per-provider per-task: `OLLAMA_LOCAL_{TASK}_TIMEOUT_SECONDS`
2. Per-provider: `OLLAMA_LOCAL_TIMEOUT_SECONDS`
3. Per-task: `{TASK}_TIMEOUT_SECONDS`
4. Global: `LLM_TIMEOUT_SECONDS`

Рекомендации по таймаутам:

- Ollama local (7B модели): 180-300 секунд для news/synthesis/verification
- Ollama local (большие модели): 300-600 секунд
- Ollama server: 180-300 секунд (зависит от сервера)
- DeepSeek API: 60-120 секунд (обычно быстрый)
- Cloud API (OpenAI, Google): 60-120 секунд

## Hugging Face прогресс и кеш

Переменные, влияющие на progress bars:
- `HF_HUB_DISABLE_PROGRESS_BARS=1` - отключает progress bars HF Hub
- `TQDM_DISABLE=1` - отключает progress bars tqdm

Для включения progress bars установите эти переменные в пустые значения или удалите их.

Кеш складывается в директорию, указанную в `HUGGINGFACE_HUB_CACHE` (по умолчанию `models/.cache/huggingface/hub`). Для очистки кеша удалите эту директорию.
